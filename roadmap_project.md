# 项目名称：Roadmap 动态科技树系统

## 1. 项目概述
本项目旨在构建一个融合个人知识管理与开发者技能图谱的 Web 应用程序。系统以技术节点为核心，展示技术间的依赖关系，并通过追踪用户的实际投入时间来动态展示技术的掌握熟练度。整体视觉交互参考 Roadmap.sh 的风格。系统支持与个人项目库形成自动化的双向链接，并在未来扩展为众包共享的技术图谱。

## 2. 核心功能需求

### 2.1 视图呈现
* **拓扑图视图：** 类似 Roadmap.sh 的依赖关系图，展示技术前置条件及层级结构。
* **看板分类视图：** 按照大类展示技术群列表。分类过程由 LLM 结合预设提示词自动完成，无需人工干预。

### 2.2 节点状态与熟练度系统
基于实际投入时间决定节点的状态，支持自定义阈值。
* **探索期：** 点亮基础节点，触发条件为浏览相关资料或查阅文档超过基础时长。
* **熟练期：** 高亮节点，触发条件为代码库中有实际代码提交且耗时达到设定阈值。
* **专精期：** 赋予特殊视觉效果，触发条件为产出长篇技术博客或完成高质量开源代码贡献。

### 2.3 双向链接与资料聚合
* **正向索引：** 点击任意技术节点，需展示所有关联的个人项目、博客文章、论文及动态记录。
* **反向查阅：** 查看具体项目详情时，需反向展示该项目所依赖及涵盖的技术栈列表。

### 2.4 众包与标准化机制
* 底层数据结构需支持多用户对同一技术实体的关联计算。
* 系统通过统计点亮同一节点的用户数量，推算出技术难度指数并反映在节点的视觉呈现上。

## 3. 系统架构与技术栈推荐

### 3.1 前端层
* **核心框架：** React 或 Vue
* **图表渲染引擎：** 推荐使用 React Flow 或 AntV G6。引擎需负责处理复杂的 DAG 图拓扑计算，并支持自定义节点内部渲染以展示用时进度条和点亮状态。

### 3.2 服务端与数据层
* **后端服务：** Python FastAPI（便于与模型脚本交互）或 Node.js。
* **数据库：** 强烈建议使用 Neo4j 或 NebulaGraph 等图数据库。利用图数据库原生支持处理节点与边的特性，能够以极高的性能解决双向反查问题。

## 4. 核心数据结构设计
建议采用如下基础数据规范，以便前后端数据交互及状态管理：

**Technology Node (技术节点)**
* `id`: 唯一标识符
* `name`: 技术名称
* `category`: 所属技术群（由 LLM 自动生成）
* `time_spent`: 累计耗时
* `status`: 当前熟练度状态标识
* `rarity_index`: 稀有度难度系数
* `resources`: 关联资料列表（包含项目链接与博文地址）

**Relation Edge (关系边)**
* `source_id`: 前置节点 ID
* `target_id`: 后置节点 ID
* `relation_type`: 关系类型（限定为依赖或关联）

**Project Node (项目实体)**
* `id`: 项目唯一标识符
* `name`: 项目名称
* `associated_tech`: 关联的技术节点 ID 列表

## 5. 自动化工作流规划
* **时间与数据采集：** 通过集成 WakaTime API 统计代码级耗时数据。
* **依赖自动分析：** 编写守护进程监控代码仓库更新，利用大语言模型自动解析项目的依赖清单及文档，提取技术节点并自动建立项目与技术的关系映射。

## 6. 开发实施路径
* **第一阶段（核心图谱 MVP）：** 完成基础的前端图表渲染，跑通本地静态 JSON 数据加载，实现拓扑图与看板视图的无缝切换。
* **第二阶段（数据层打通）：** 搭建后端与图数据库，实现项目依赖解析脚本，并接入 LLM 自动分类接口。
* **第三阶段（动态系统与众包）：** 完善时间阈值计算与状态进阶逻辑，加入用户统计模型与众包难度系数算法。
